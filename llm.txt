MazadAuto — Enchères Automobile en Tunisie
=============================================

URL: https://mazadauto.tn
Stack: Next.js 16.2.6 (Turbopack), TypeScript strict, Tailwind CSS v4, Framer Motion
Base de données: Supabase (PostgreSQL + Auth + Realtime + Storage)
Notifications: Resend (emails), Twilio (SMS), Supabase Realtime (in-app)
CI/CD: Vercel

Architecture
------------
- App Router (Next.js 16). 42 pages + API routes dynamiques.
- `app/(marketing)/` — Landing page (Navbar, Hero, Live Auctions, How It Works, Trust, CTA, Footer)
- `app/(auth)/` — Connexion, inscription (acheteur/vendeur), wizard 5 étapes
- `app/encheres/` — Listing avec filtres, tri, pagination (client-side, URL search params)
- `app/encheres/[slug]/` — Fiche véhicule (SSG), OG image dynamique, JSON-LD, photos, inspection, documents PDF
- `app/dashboard/` — Dashboards acheteur/vendeur/admin connectés à Supabase
- `app/api/og/` — Open Graph image dynamique 1200×630 PNG
- `app/sitemap.ts`, `app/robots.ts` — SEO
- API routes publiques: `/api/vehicles`, `/api/vehicles/[slug]` pour contourner RLS sur photos/documents/inspection
- API routes admin: `/api/admin/stats`, `/api/admin/kyc-pending`, `/api/admin/users`
- API route auth: `/api/me` pour récupérer le profil utilisateur

Données
-------
- 10 véhicules actifs en base avec 16 photos Supabase Storage, 10 PDF contrôle technique, 60 items d'inspection
- 3 utilisateurs de test (admin, vendeur, acheteur) avec KYC vérifié et portefeuilles actifs
- Commission acheteur 2% (min 200 DT), vendeur 0%. Caution 200 DT pour enchérir.

Authentification & KYC
----------------------
- Inscription: email/password via Supabase Auth.
- KYC: upload CIN (recto/verso), selfie, RIB, patente/statuts selon type de compte.
- Validation manuelle par admin (24-48h).
- Caution bancaire 200 DT requise pour enchérir.

Mode Test (E2E)
----------------
- Bouton "Mode Test" dans la navbar du landing page.
- Active un cookie `__test_mode=true` + localStorage.
- Bypass complet de l'authentification dans le middleware pour `/dashboard/*`.
- Peut être supprimé sur demande.

Filtres enchères (URL search params)
------------------------------------
- Marque (radio, liste dynamique depuis les données)
- Carburant: essence | diesel | hybride | electrique | gpl
- Boîte: manuelle | automatique
- Prix min/max (en milliers de DT, filtre sur current_price)
- Kilométrage max
- Année min
- Tri: plus récentes / se terminent bientôt / prix croissant / décroissant

SEO
---
- Metadata root avec OG/Twitter/JSON-LD.
- `generateMetadata` dynamique sur chaque fiche véhicule.
- `/api/og` génère une image 1200×630 avec titre + prix.
- Sitemap dynamique depuis Supabase, robots.txt.
- Structured data: Car + Organization schema.

Production Hardening
--------------------
- Zod validation sur auth, vehicle, auction, bid.
- Security headers dans middleware (X-Frame-Options, HSTS, CSP, etc.).
- Validation uploads: MIME whitelist, 5MB max, magic bytes.
- Playwright E2E: 38 tests passants (landing, browsing, auth, dashboard, bidding, security, real login).
- Build: 42 pages, 0 erreurs TypeScript.

Conventions
-----------
- Pas de UI library (shadcn/MUI) — composants custom Tailwind uniquement.
- Couleurs: fond blanc `#ffffff`, accent vert `#16a34a`, typographies Bricolage Grotesque + Inter.
- Mobile-first, fully responsive.
- Tout le texte en français, prix en DT (milliers).
- `data-testid` sur les éléments critiques pour E2E.

Points d'attention connus
-------------------------
- RLS `profiles` : la policy `admin_all_profiles` provoque une récursion infinie. Contournement en place : toutes les requêtes `profiles` passent par le service role (`createAdminClient`) côté serveur, ou par des API routes (`/api/me`, `/api/admin/*`) pour le client.
- Migration `011_fix_profiles_rls_recursion.sql` créée mais non appliquée (besoin d'accès psql/CLI Supabase).
- Cron job `finalize-ended-auctions` tourne toutes les minutes en production. Pour les tests, les dates des enchères doivent être dans le futur.
- React Email est remplacé par des templates HTML string (`lib/email/templates.ts`).
- `date-fns` est remplacé par un helper `timeAgo()` custom.
- Upstash rate limiting et Sentry sont des placeholders commentés.
